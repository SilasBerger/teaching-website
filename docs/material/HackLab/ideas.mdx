# Ideas
## Claude
1. Hidden in Plain Sight: A flag is stored in an HTML comment somewhere on the page. Students need to "View Page Source" or inspect the DOM.
2. Invisible Ink: A `<div>` containing a flag is styled with color: white; background: white (or display: none). Students use the Inspector to find and read it, or select-all the page.
3. The Forbidden Button: A "Submit" button has a disabled attribute. The flag is revealed only when the form is submitted. Students use the Inspector to remove the disabled attribute and click it. Submission just reads a hardcoded value client-side — no real backend needed.
4. Metadata Mole: A flag is hidden in a `<meta>` tag in the `<head>` — something like `<meta name="secret" content="FLAG{...}">`. Not visible in the rendered page at all.

5. Console Confession: A flag is printed to the browser console via console.log() on page load, alongside other noise. Students must open the DevTools console to find it.
6. Source Code Spelunker: A flag is stored in a JS variable in a `<script>` block (or a loaded .js file), visible in the Sources tab. Students must read the source — not just the DOM.
7. The Timed Reveal: A flag appears in the DOM for 200ms on page load (via setTimeout removal), then disappears. Students can catch it by pausing execution with a breakpoint in DevTools, or by reading the JS source.
8. Caesar's Secret: A ciphertext is displayed on the page. The flavour text hints at ancient Rome. Students apply a Caesar cipher (ROT13 or a custom shift) to decode the flag. Tools like CyberChef are acceptable.
9. Not-So-Secret Encoding: A flag is Base64-encoded and displayed as a string. The flavour text says something like "this isn't encryption." Students decode it — either using CyberChef, an online tool, or the browser console (atob()).
10. The Key Is On The Page: A flag is encrypted with a simple XOR or Vigenère cipher. The key is hidden elsewhere on the page (e.g. in an HTML comment or a CSS custom property like --hint: "keyword"). Students must find the key and apply it.
11. Hashed Out: A SHA-256 hash of a short, guessable word is displayed (e.g. the hash of "apple" or "switzerland"). The flavour text says "I remember the answer, I just forgot the question." Students look up the hash on a site like CrackStation or brute-force common words. The flag is `FLAG{<the word>}`.
12. The Invisible Request: On page load, the page silently makes a fetch() to a public echo/mock API (e.g. `httpbin.org/get?flag=FLAG{...}`). The flag is in the request URL or response body, visible only in the Network tab.
13. Request Headers: The page sends a fetch with a custom request header (e.g. `X-Secret-Flag: FLAG{...}`). Students find it in the Network tab under the request headers of that call.
14. Cookie Crumble: On page load, a flag is written into a cookie via document.cookie. Students find it in DevTools → Application → Cookies, or by typing document.cookie in the console.
15. The Locked Door: A fake "admin login" form is on the page. The JS validation compares the entered password against a hardcoded plaintext string in the source. Students read the source to find the password; entering it reveals the flag. (No real authentication — purely client-side.)
16. localStorage Locker: On page load, a flag is written to localStorage. It's never displayed on screen. Students find it via DevTools → Application → Local Storage, or via localStorage.getItem() in the console.
17. The Robots Riddle: A robots.txt file exists at the site root with a Disallow: /ctf/supersecret entry. That path contains the flag. The flavour text says "search engines know where not to look." This introduces the concept of robots.txt as a real-world recon technique.
18. The Polyglot Image:
A flag is hidden in the alt text, title attribute, or EXIF-style `<figure>` caption of an image — not visible unless you inspect the element. Alternatively, the flag is in a CSS content property on a ::before pseudo-element, which appears visually but isn't in the HTML source directly.
